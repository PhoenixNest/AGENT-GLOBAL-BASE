# Gemini CLI Ecosystem Specification

**Status:** Operational Governance Document
**Target Workspace:** `agent-global-base`

## Lifecycle Status Types

To govern the evolution of the multi-agent swarm, all items in this specification (Sub-agents, Skills, and Hooks) must utilize one of the following official lifecycle status types:

- **`Reference`**: Existed in a legacy "before configuration" state; acting as a reference point before implementation.
- **`Draft`**: Currently being written, designed, or reviewed.
- **`Testing`**: Deployed to the `.gemini/` configuration ecosystem but currently undergoing evaluation and testing.
- **`Migrated`**: Successfully processed, appropriately named (`[Domain]-[Subdomain]-[Role]-[Name]`), and transitioned into the live `.gemini/` configurations.
- **`Active`**: Fully deployed, verified, and in production use by the swarm orchestrator.
- **`Deprecated`**: Phased out or superseded by a newer implementation, but retained for historical pipeline compatibility.
- **`Archived`**: No longer invoked by the orchestrator and safely removed from active execution paths.

## Ecosystem Architecture Strategies

The elements mapped in this report follow two core design strategies inside the `.gemini/` configuration ecosystem to maximize scalability and discoverability without overloading the Context Window or the CLI's resolution mechanics.

### 1. Progressive Disclosure (The Router/Reference Skill Pattern)

Instead of dumping hundreds of skills into a single flat `.gemini/skills/` directory—which causes CLI naming collisions and impossible discoverability—we utilize a **Router/Reference Pattern** grouped by explicit architectural domains (e.g., `android-engineering`, `quality-assurance-and-testing`).

- **The Router:** Each domain contains a single, lightweight `SKILL.md` file at its root (e.g., `.gemini/skills/android-engineering/SKILL.md`). The Gemini CLI cleanly discovers this single "Super-Skill."
- **The References:** The router acts as a dispatcher. It dynamically loads massive, highly specific sub-skills from its `references/` subdirectory (e.g., `references/jetpack-compose.md`) _only_ when the context demands it. This keeps the CLI's `activate_skill` list short while providing deep, categorized intelligence.

### 2. Synchronized Agent Competencies

Every employee profile is treated as a full-fledged, invokable Sub-agent within the `.gemini/agents/` directory. Because skills are now explicitly categorized by domain, we can synchronize each agent's profile with their specific job-related competencies.

- When you invoke an agent like `chief-design-officer`, its profile is explicitly linked to the exact skills it owns within the domain structure (e.g., `` `product-design/design-to-engineering-handoff` ``). This creates a tightly integrated, easily readable ecosystem where every agent knows exactly what skills they possess.

### 3. Important Implementation Requirements

To ensure the hooks system operates correctly with the Gemini CLI, the following implementation details are strictly required:

- **Registration in `settings.json`:** Moving scripts to `.gemini/hooks/` does not automatically activate them. They must be explicitly registered under the corresponding event type in `.gemini/settings.json` (e.g., `"SessionStart"`, `"BeforeAgent"`, etc.).
- **Strict JSON I/O:** Hooks must be completely silent on standard output (`stdout`) except for their final valid JSON payload. Any logging or progress updates must be directed to standard error (`stderr`).
- **Exit Codes:** Scripts should return `0` on success, `2` to block an action, or a non-zero code to indicate a non-fatal warning.

### 4. Hook Implementation Strategy (The Wrapper Pattern)

To preserve the general design philosophy and reusability of the `core-component-00` architecture, **we must not modify the original Python scripts** to accommodate the Gemini CLI's specific standard I/O JSON protocol.

Instead, the `.gemini/hooks/` directory must utilize a **Wrapper Pattern**.

For each required hook, a thin wrapper script should be created inside `.gemini/hooks/` (e.g., `.gemini/hooks/context_assembler_wrapper.py`). This wrapper will:

1. **Import** the unmodified core logic from `core-component-00` (e.g., `from core_component_00.context_engineering.implementations.context_assembler import ContextAssembler`).
2. **Read** the incoming JSON payload from `sys.stdin` as provided by the Gemini CLI.
3. **Execute** the core component's classes and functions.
4. **Route** all logging, debugging, or progress output to `sys.stderr` to ensure the JSON parser is not broken.
5. **Output** the final, strictly formatted JSON response to `sys.stdout`.
6. **Exit** with the appropriate status code (`0` for success, `2` for blocking/rejection, or other non-zero codes for warnings).
7. **Idempotency & Performance:** Since events like `SessionStart` fire for every sub-agent invocation, wrappers for expensive operations (e.g., RAG initialization) should implement state checks (e.g., checking for a shared lock file or a pre-warmed cache) to avoid redundant, high-latency initializations across sessions.

This approach ensures that `core-component-00` remains a pure, decoupled reference methodology, while the `.gemini/hooks/` wrappers handle the proprietary integration layer for the Gemini CLI.
